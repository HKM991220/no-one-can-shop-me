# 热更新发布维护指南

本文档对应项目内 `HotUpdateService` 和 `scripts/hotupdate/build-manifest.js`。

## 1. 一次性接入检查

- 运行时入口：
  - `assets/Mian.ts` 在启动时恢复 `searchPaths`
  - `assets/common/ui/UIService.ts` 在 Loading 阶段执行热更新
- 本地 manifest（内置包）：
  - `assets/resources/hotupdate/project.manifest`
  - `assets/resources/hotupdate/version.manifest`
- 发布脚本：
  - `scripts/hotupdate/build-manifest.js`

## 2. 首次配置

1. 复制配置模板：
   - 将 `scripts/hotupdate/config.template.json` 复制为 `scripts/hotupdate/config.json`
2. 修改配置：
   - `platforms`：本次要生成的目标平台（如 `android`、`ios`）
   - `buildDirByPlatform`：平台对应构建目录
   - `outputDir`：manifest 产物目录（会按平台拆分）
   - `releaseDir`：归档发布包目录（会按版本和平台拆分）
   - `packageUrlByPlatform`：各平台 CDN 地址
   - `version`：本次发布版本号（建议语义化）
3. 修改内置 manifest 的 URL（`assets/resources/hotupdate/*.manifest`）为正式地址。

## 3. 每次发版流程

1. 本地打包原生资源（Android/iOS）。
2. 更新 `scripts/hotupdate/config.json` 中的 `version`。
3. 运行命令生成 manifest：
   - `npm run hotupdate:manifest:config`
4. 将目标平台 manifest 同步到内置资源（用于下次安装包）：
   - Android: `npm run hotupdate:embed:android`
   - iOS: `npm run hotupdate:embed:ios`
5. 上传文件到 CDN（目录建议按版本隔离）：
   - 新版本资源文件（`src/`、`assets/` 等）
   - `project.manifest`
   - `version.manifest`
6. 验证：
   - 安装旧包启动，观察是否进入检查更新并下载
   - 更新后自动重启，版本资源生效

## 4. 一键命令（Android）

- 生成 manifest + 同步内置 manifest：
  - `npm run hotupdate:release:android`

产物目录示例：

- 平台 manifest：
  - `build/hotupdate-manifest/android/`
  - `build/hotupdate-manifest/ios/`
- 归档可发布目录：
  - `build/hotupdate-release/{version}/android/`
  - `build/hotupdate-release/{version}/ios/`

## 5. 线上维护建议

- 版本号单调递增（避免“同版本内容变化”）。
- `version.manifest` 先发，资源和 `project.manifest` 后发，避免空窗。
- CDN 建议开启缓存控制：
  - `version.manifest` / `project.manifest` 低缓存或强制回源
  - 资源文件可长缓存（文件名变动由打包控制）
- 保留至少 1 个稳定旧版本以便快速回滚。

## 6. 回滚策略

1. 将 CDN 上的 `version.manifest` 与 `project.manifest` 指回稳定版本。
2. 如需彻底回退，连同资源目录一起回滚。
3. 客户端下次检查更新会拉取回滚后的版本定义。

## 7. 常见问题

- **只在原生端生效吗？**
  - 是。Web/编辑器会自动跳过热更新流程。
- **更新失败怎么办？**
  - `HotUpdateService` 已内置失败重试（默认 3 次）。
- **为什么更新后要重启？**
  - 热更新替换资源搜索路径后需重启才能完整生效。
